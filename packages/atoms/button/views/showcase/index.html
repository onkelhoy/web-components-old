Overview Buttons are essential elements of UI Design. They drive the user flow forward and enable the user to take actions.
Color variants
  1. Primary: used for the main action of a page, such as Submit or Sign Up. It should be used sparingly.
  2. Secondary: for important actions that are not considered primary.
  3. Inverse: used on inverse backgrounds where primary and secondary buttons do not have enough contrast.
  4. Danger, Warning, & Success: used to communicate danger, warning, and success messages.
  5. Disabled: for disabled buttons that cannot be interacted with.
Style variants
  1. Filled: for important actions, where user attention is very important.
  2. Outlined: for medium-important actions.
  3. Clear: for the least important actions.

Focused When focusing on a button using the keyboard or voice controls, the state changes to the focused state. Focused buttons have a focus ring around them which matches the corner radius. While in the focused state, buttons can also have a hover or pressed state. The focus ring uses the color --o-color-border-strong, has a 0.125rem (2px) width, and has 0.0625rem (1px) margin between itself and the button. The focus ring is formatted outside the button and should not affect the button’s vertical placement on a page. Disabled buttons are not able to be focused.

Loading Buttons can indicate that indeterminate progress is taking place by using the loading state. When a button is loading, the button label and icons are hidden and a spinner is shown in its place, while maintaining the width that it would have if the label and icons were visible. Use the loading state sparingly. It should be reserved only for when the progress is expected to be quick (taking 5 seconds or less), and when there is no better way to communicate progress. For determinate progress or interment progress expected to take more than 5 seconds, use a progress bar.

Alignment Button alignment depends on where they appear and if they are nested in another component.
Left-aligned Generally, the primary button is aligned to the left and stays close to where the user’s focus is. This applies to forms and cards.
Right-aligned When nested within dialogs and wizards, the primary button is traditionally aligned to the right. This also applies to other components that a button is nested in, such as in-line notifications.
Center-aligned Buttons can be center-aligned when the rest of the design is also center aligned.
Full-width In some cases, buttons can span the entire width of a container. This usually happens within cards, side panels, and dialogs.

Right-aligned dialogs, wizards, in-line notifications, messages

Visual Weight If there are more than 3 CTAs on a page, it’s recommended to use outlined or clear buttons for the majority of instances. This is due to the visual weight of filled buttons, which can appear too heavy if there are many of them.
Visual weight
Icons Icons should only by used in buttons if additional clarity is required. Icons should not be used as decoration. If a button does have an icon, the icon may be a leading icon or a trailing icon (aka prefix an suffix in dev environments). The default position for icons is leading. Trailing icons may only be used if it doesn’t make sense for the icon to be leading. Icon buttons, buttons that only have an icon and no label, may also be used, but must be accompanied by a tooltip on hover.

Labels Case Use sentence case for button labels. Don’t use all-caps for emphasis. Verbs Buttons represent actions and the best way to represent an action in writing is with a verb. Labels written as nouns or adjectives can be disorienting. Clarity Make sure that the label clearly states the outcome of an action and uses the same words as other buttons in the experience that do the same thing. Tone Be aware of tone and avoid decorations like emojis or exclamations.

This may occur when switching the displayed language. The icon is vertically aligned to the center in these cases. The maximum width of the button is determined on a contextual basis. Combinations A major CTA can be combined with one or more minor CTAs. Select the minor CTA type based on importance. Make sure all minor CTA’s within a button group are the same. Multiple minor CTAs can be combined. See the alignment section above for guidelines on whether to use the right or left aligned button group variants.

Button Width Within a stacked button group, buttons should all be the same width. To do this, keep the button with the most width on “Hug” and match the width of the others using “Fill”. Horizontal button groups may have buttons of varying widths. When aligning button groups to the grid, each group should be treated as an individual component.

Danger, Warning & Success Use a button group with a danger button to indicate destructive actions, a warning button to indicate potentially destructive actions, or a success button to indicate successful actions. Only one danger, warning, or success button may be used per group, and only the “filled” variant may be used. The minor buttons in the group must be “outlined-secondary”.

Icons Icons may be used in groups with one of the following formats:
  1. Only the major button in the group has an icon
  2. All buttons in a group have an icon
  3. No buttons in a group have an icon
Icons in button groups must either all be leading or all be trailing. In most cases they will all be leading.
